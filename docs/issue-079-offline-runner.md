# ISSUE_079 offline runner

The offline runner makes the validation workflow executable before a licensed
historical-data package is purchased. It uses deterministic demo rows by
default, or a user-supplied CSV with the same explicit point-in-time contract.
Demo output is always classified as **Inconclusive** and must not be used as a
production or marketing performance claim.

## Run immediately

```bash
PYTHONPATH=. python -m codes.workers.issue_079_backtest \
  --config configs/issue_079.demo.json \
  --output-dir artifacts/issue-079
```

The command writes `issue-079-result.json` and `issue-079-report.md`.

## Test the evaluator with known ground truth

Generate controlled data and run the same evaluator four ways:

```bash
for scenario in strong null inverted risk-screen; do
  PYTHONPATH=. python -m codes.workers.generate_issue_079_data \
    --scenario "$scenario" \
    --output "/tmp/issue-079-${scenario}.csv"
  PYTHONPATH=. python -m codes.workers.issue_079_backtest \
    --input "/tmp/issue-079-${scenario}.csv" \
    --output-dir "artifacts/issue-079/${scenario}"
done
```

Expected diagnostics:

- `strong`: `strong-synthetic-signal`
- `null`: `null-or-weak-synthetic-signal`
- `inverted`: `inverted-synthetic-signal`
- `risk-screen`: usually `mixed-or-insufficient-signal`, because it is designed to separate downside protection from outperformance

These scenarios validate the evaluator's behavior. They do not prove the real
algorithm is good because the scores and outcomes are synthetic. To evaluate
the real algorithm, replace the CSV with point-in-time scores and independently
computed future outcomes from the licensed data pipeline.

## Null test using the real composite scorer

The `random-algorithm` scenario generates random factor inputs, calculates the
score through the existing `codes.engine.factor_backtest._score_with_weights`
path, and then generates independent random future returns. Run many seeds:

```bash
PYTHONPATH=. python -m codes.workers.issue_079_randomized_trial \
  --count 100 \
  --output artifacts/issue-079/randomized-null-trial.json \
  --markdown-output artifacts/issue-079/randomized-null-trial.md
```

Interpretation: the mean high-score-minus-low-score spread should be near zero,
and its 95% confidence interval should include zero. A single seed may look
excellent or terrible by chance; it must never be treated as evidence.

The Markdown output is intentionally publication-oriented: it shows the
result, lucky and failed runs, exact reproduction details, and limitations in
the same document. It should be presented as a transparency/control-test
report, not as proof of investment performance.

## CSV handoff contract

Required columns are:

```text
symbol,analysis_date,available_at,execution_date,score,signal,
start_price,end_price_1y,spy_start_price,spy_end_price_1y
```

Optional diagnostic columns are `sector`, `market_cap_band`, `regime`,
`survivor_status`, and any `model_<name>` contribution columns. The signal
inputs must be available on or before `analysis_date`; forward outcome columns
must be generated by a separate later-period evaluation step.

```bash
PYTHONPATH=. python -m codes.workers.issue_079_backtest \
  --input /path/to/point-in-time-predictions.csv \
  --config configs/issue_079.demo.json \
  --output-dir artifacts/issue-079/run-001
```

The current phase intentionally does not claim complete 20-year coverage,
confidence intervals, rolling 5/10-year outcomes, production database
persistence, or a public Validation Center. Those require the licensed data
ingestion and immutable run schema planned for the next phase.
